# Development Plan

Деталізація [ROADMAP.md](ROADMAP.md) до задач, кожна з яких розрахована максимум на один робочий день.

Формат: Фаза → Модуль → Задачі. Задачі йдуть у логічному порядку виконання всередині модуля. Задачі, що вимагають архітектурного чи продуктового рішення (а не лише реалізації), позначені словами "Ухвалити рішення" — це усвідомлено, бо в поточних документах ([ARCHITECTURE.md](ARCHITECTURE.md), [DATABASE.md](DATABASE.md)) частина цих речей не визначена.

Код не описується — лише задачі.

Позначки прогресу: ⬜ не почато, 🟡 частково (код-сторона готова, лишились дії поза репозиторієм — зовнішні акаунти, ручне налаштування), ✅ виконано.

---

## Phase 0 — Фундамент

Передує Phase 1 з ROADMAP.md. Без цього подальші фази неможливо оцінити коректно.

✅ Ухвалити технологічний стек (backend, frontend, база даних, хостинг, real-time шар) і зафіксувати рішення в ARCHITECTURE.md

✅ Деталізувати DATABASE.md: додати поля, типи даних і зв'язки (1:1, 1:N, N:N) для всіх 16 наявних сутностей

✅ Додати в DATABASE.md сутності, яких бракує для покриття ROADMAP: Bookmark, DiaryEntry, Listing/Transaction, Role/Permission (Role/Permission спрощено до поля `role` в User)

✅ Звести єдину мапу "Фаза ROADMAP → Модуль ARCHITECTURE → Сутності DATABASE", щоб усунути розбіжності між трьома документами

✅ Спроектувати механізм приватності/прозорості даних (журнал згод, експорт даних користувача, видалення акаунта) відповідно до Principle 4 і 5

✅ Налаштувати структуру репозиторію, лінтери, форматери, pre-commit hooks

✅ Налаштувати CI (автоматичні тести й білд на кожен pull request)

🟡 Налаштувати CD на стейджинг-середовище — код-сторона готова (`railway.json`, health-check, [DEPLOYMENT.md](DEPLOYMENT.md)), ручні кроки (GitHub-репозиторій, Railway-акаунт) ще не виконані

✅ Перевести DESIGN_SYSTEM.md у конкретні дизайн-токени: hex-коди для Electric Blue/Amber/Red/Green, типографічна шкала, spacing-шкала, брейкпоінти

✅ Побудувати базові UI-компоненти (кнопка, інпут, картка, модалка, toast) згідно з дизайн-токенами

✅ Підготувати інфраструктуру автентифікації (сховище секретів, бібліотека хешування паролів, сховище сесій/токенів)

## Phase 1 — Auth, Профіль, Пошук

### Авторизація

✅ Спроектувати схему User (поля для логіну: email, password_hash, статус верифікації, timestamps)

✅ Визначити контракт API для входу (login)

✅ Реалізувати перевірку credentials і хешування пароля

✅ Реалізувати логаут (інвалідація сесії/токена)

✅ Реалізувати захист від brute-force (rate limiting на auth-ендпоінтах)

✅ UI: екран входу (форма + валідація полів)

✅ UI: стани помилок (невірний пароль, заблокований акаунт, непідтверджений email)

✅ Написати тести для login flow (успіх, невірний пароль, заблокований акаунт)

### Реєстрація

✅ Визначити контракт API для реєстрації

✅ Реалізувати правила валідації email і складності пароля (+ формат username)

✅ Реалізувати захист від дублікатів email (+ username, вимога контракту API.md)

✅ Реалізувати генерацію токена підтвердження email і відправку листа

✅ Реалізувати ендпоінт підтвердження email

✅ Реалізувати запит на скидання пароля (endpoint + лист)

✅ Реалізувати підтвердження нового пароля

✅ UI: форма реєстрації

✅ UI: екран "підтвердіть email"

✅ UI: форми скидання пароля (запит і встановлення нового)

✅ UI: чекбокс згоди при реєстрації з посиланням на політику приватності (Principle 4/5)

### Профіль

⬜ Визначити додаткові поля профілю, яких немає в поточному DATABASE.md (bio, avatar, локація тощо)

⬜ Визначити, які поля профілю публічні, а які приватні (Principle 4)

⬜ Реалізувати ендпоінт перегляду профілю (власного і чужого, з різним набором полів)

⬜ UI: екран перегляду власного профілю

⬜ UI: екран перегляду чужого профілю (публічна версія)

### Редагування профілю

⬜ Реалізувати ендпоінт оновлення профілю

⬜ Ухвалити рішення щодо сховища для завантажених файлів (аватар) і реалізувати завантаження

⬜ UI: форма редагування профілю з валідацією

⬜ UI: завантаження і попередній перегляд/обрізка аватара

⬜ Реалізувати видалення акаунта (з підтвердженням) — Principle 5

⬜ Реалізувати експорт власних даних користувача — Principle 5

### Пошук користувачів

⬜ Реалізувати ендпоінт пошуку користувачів за іменем/username

⬜ Налаштувати індексацію для швидкого текстового пошуку

⬜ UI: екран пошуку зі списком результатів
