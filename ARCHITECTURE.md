### Розбіжності, які лишаються відкритими

- **Навчальні й голосові кімнати** (Phase 5) не мають ні окремого модуля в ARCHITECTURE, ні сутностей у DATABASE.md — концепція взагалі не спроєктована (позначено `TODO` в DEVELOPMENT_PLAN.md).
- **Відеодзвінки** (Phase 6) так само без сутності `Call` у DATABASE.md — окрема майбутня задача.
- **Feed** — не окремий крок ROADMAP, а технічна реалізація Phase 3 (стрічка постів), не нова сутність.
- **Notifications** і **Settings** — наскрізні модулі: Settings починається в Phase 1 (редагування профілю), Notifications стає потрібним з Phase 2 (запити дружби, повідомлення), обидва ростуть далі — не привʼязані до однієї фази.
- **Admin Panel** — немає жодної відповідної фази в ROADMAP.md. Наразі покривається лише полем `User.role` (DATABASE.md); повноцінного UI/API не заплановано жодною задачею DEVELOPMENT_PLAN.md — свідома прогалина, не помилка.

## Приватність і прозорість даних

Продовження [DEVELOPMENT_PLAN.md](DEVELOPMENT_PLAN.md) → Phase 0 → "Спроектувати механізм приватності/прозорості даних". Реалізує Principle 4 (Конфіденційність) і Principle 5 (Прозорість) з [PRINCIPLES.md](PRINCIPLES.md).

### Журнал згод

Сутність `ConsentRecord` ([DATABASE.md](DATABASE.md#consentrecord)) фіксує, на яку версію якого документа (умови використання, політика приватності) і коли погодився користувач. Показується користувачу за запитом — "на що я погодився і коли".

### Журнал чутливих дій

Сутність `AuditLog` ([DATABASE.md](DATABASE.md#auditlog)) — мінімальний скоуп, лише дії навколо персональних даних: запит експорту, запит видалення акаунта, зміна пароля. Не повний audit trail кожної дії в системі — це було б over-engineering для MVP.

### Експорт даних

За запитом користувача — структурований (JSON) дамп усіх сутностей, де він є власником даних: профіль (User), Goal, Progress, DiaryEntry, Post, Comment, надіслані Message, Achievement/Badge, Bookmark. Генерується на вимогу, не зберігається окремою сутністю — сам факт запиту фіксується в AuditLog.

### Видалення акаунта

Дані поділяються на дві категорії за типом впливу на інших користувачів:

- **Суто приватні** (Goal, Progress, DiaryEntry, зміст Message) — видаляються остаточно.
- **Публічний внесок у спільний контекст** (Post, Comment) — анонімізуються: посилання на автора замінюється на "видалений користувач", сам контент лишається, щоб не ламати треди для інших користувачів.
- User — soft-delete через `deleted_at`, остаточне видалення через N днів.

Тривалість "N днів" до остаточного видалення теж не визначена. Потребує підтвердження, якщо зʼявляться юридичні вимоги конкретної юрисдикції.

### Свідомо відкладені рішення

Не вирішуються зараз, кожне — окрема задача відповідної фази з DEVELOPMENT_PLAN.md:

- SFU для голосових/відео кімнат (Phase 5-6)
- LLM-провайдер для AI Помічника (Phase 6)
- Платіжний провайдер для Marketplace знань (Phase 6)
- Фреймворк мобільного застосунку (Phase 6)
- **Email-провайдер для транзакційних листів (Phase 1)** — виявлено під час реалізації верифікації email/реєстрації. На відміну від решти пунктів списку, це вже потрібно зараз для production; локально й у тестах `lib/email.ts` виводить лист у консоль замість реальної відправки, щоб не блокувати розробку. Production-деплой не зможе відправляти листи, поки провайдера не обрано й не підключено.
- **Сховище для довільних не-зображень (документи тощо)** — Cloudinary (вище) закриває аватари й медіа-зображення, але не є універсальним object storage. Якщо пізніше знадобиться зберігати файли не-медіа типів (наприклад, вкладення в повідомленнях) — окреме рішення (S3-сумісне: R2/S3) буде потрібне тоді.
- **Ручний crop аватара** — свідомо не реалізовано в UI. Cloudinary вже центрує/кропає під обличчя (`gravity: face`) до 512×512 на сервері; для стандартних фото цього достатньо. Якщо продуктово знадобиться ручний вибір рамки (drag/zoom перед заливкою) — окрема майбутня задача, ймовірно з бібліотекою на кшталт `react-easy-crop`.

### Відомі ризики

- WebSocket-сервіс — виняток з "один проєкт": Messaging (Phase 2) вимагає окремого процесу поруч із Next.js.
- Схема Prisma напряму кодує DATABASE.md, включно з позначеними там TODO-припущеннями (Achievement/Badge, поліморфні Reaction/Notification) — їх варто звірити з продуктовим баченням до першої міграції.
- Власна автентифікація — більше контролю й відповідності Principle 4/5, але вся відповідальність за безпеку (password reset, ротація токенів, захист від brute-force) на нас.
- Найдешевші тарифи Railway/Render мають обмеження ресурсів і можуть "засинати" — прийнятно для Phase 1-2, Phase 5-6 (відео, голос, AI) майже напевно вимагатимуть апгрейду плану.
- Prisma 7 вимагає driver adapter замість прямого рядка підключення — `PrismaClient` конструюється через `@prisma/adapter-pg` (`lib/prisma.ts`), а не `new PrismaClient({ datasourceUrl })`, як у попередніх версіях. Виявлено під час налаштування репозиторію, не було відомо на момент вибору стеку.
- Rate limiting на `/api/auth/login` ([lib/rate-limit.ts](lib/rate-limit.ts)) — in-memory, не Redis. Скидається при рестарті процесу й не працює коректно при кількох інстансах Railway (кожен матиме власний лічильник). Прийнятно для одного інстансу зараз; апгрейд на Redis — коли BullMQ (Tech Stack) все одно приведе Redis у проєкт.
- **Перший реальний деплой на Railway провалився** через `pnpm-workspace.yaml`: файл, автоматично створений локальним `pnpm approve-builds` для дозволу build-скриптів (`argon2`, `@prisma/engines`, `prisma`, `sharp`, `unrs-resolver`), не мав поля `packages:`. Сама наявність цього файлу вмикає "workspace-режим" pnpm, який без `packages:` падає з `ERROR packages field missing or empty` — локальна версія pnpm це якось пропускала, Railway-оточення ні. Виправлено додаванням `packages: ["."]`. Урок: `pnpm-workspace.yaml` не можна створювати "напівавтоматично" без ручної перевірки — CI/PaaS-середовища можуть бути строгішими за локальне.
- Заливка аватара (`POST /api/users/me/avatar`) читає весь файл у пам'ять перед перевіркою розміру — Next.js App Router Route Handlers не мають вбудованого ліміту тіла запиту на рівні платформи. 5MB-ліміт застосунку рятує від зловживання розміром відповіді, але не від тимчасового буферизування файлу в пам'яті процесу під час самого запиту. Прийнятно при малих файлах-аватарках і одному інстансі; варто переглянути, якщо цей же шлях почне приймати більші файли (медіа в постах).
